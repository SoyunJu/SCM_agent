# 사용자 선호 코드 제공 방식

> 지금까지의 대화에서 도출한 코드 제공 및 제안 방식 정리

---

## 소통 방식

- **한국어**로 짧고 직접적으로 답변
- 불필요한 서문("~하겠습니다", "~살펴보겠습니다") 없이 바로 본론
- 완료 후에는 무엇을 했는지 한 줄 요약만

---

## 설명 방식

- **아키텍처 도식/상세 설명 반복 금지** ("아키텍처 매번 제시할 필요 x")
- 원인 → 간단한 설계 원리 → 구현 코드 순서
- 예: "원인: `StringIO`는 encoding 무시됨 → BytesIO + BOM으로 수정"처럼 한 줄 원인 + 코드

---

## 코드 제공 방식

- **배치 처리** — 여러 이슈를 묶어서 한 번에 수정
- **포괄적 수정** — "전체적으로 소문자 말고 대문자로" 처럼 코드베이스 전반 일관성 있게 반영. 한 곳만 고치는 반쪽짜리 수정 금지
- **근본 원인 수정** — 임시방편(workaround) 금지. 문제의 원인을 찾아서 정확히 수정
- **헬퍼 패턴 우선** — 반복되는 비교/변환 로직은 헬퍼 함수로 추출 (`norm()`, `normSev()` 등)
- 수정 시 영향 범위가 넓으면 관련 파일 전체를 찾아서 일괄 수정

---

## 제안 방식

- **플랜 먼저, 실행은 승인 후** — 복잡한 작업은 계획을 제시하고 승인 대기
- 하지만 단순하거나 명확한 작업은 설명 없이 바로 실행 가능
- "그냥 해줘" 식으로 직접 지시하면 확인 없이 실행
- 대안 나열보다 **권장 방향 하나**만 제시

---

## Git / 브랜치 규칙

- 작업 완료 후 **반드시 지정 브랜치로 push** (`develop` 또는 `claude/...`)
- 커밋 메시지는 영어, 변경 내용 핵심만 한 줄
- PR 생성은 명시적으로 요청 시에만

---

## 금지 사항

| 금지 | 이유 |
|------|------|
| 매 답변마다 아키텍처 도식 재설명 | 이미 알고 있음 |
| 중간에 "이렇게 해도 될까요?" 확인 질문 남발 | 판단은 직접 해서 제시 |
| 소문자/대문자 섞인 enum 비교 | 버그 원인이 됨 |
| `StringIO` + encoding 파라미터 | 실제 encoding 무시됨 |
| `useEffect` + manual fetch 패턴 | React Query 사용 |
| 단일 파일만 고치는 반쪽 수정 | 전파형 수정 필요 |

---

## 선호 패턴 요약

```
이슈 접수
  → 원인 한 줄 설명
  → 수정 코드 (전체 영향 범위 반영)
  → 커밋 & push
  (아키텍처 설명 생략)
```
